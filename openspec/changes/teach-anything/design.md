## Context

当前系统是 Git 教学仓库，教学流程（TEACHING_PROTOCOL.md 的 7 步流程）与 Git 模块结构（module-0 至 module-6）深度耦合。用户若想学其他技能（如 Python、Docker、Git 高级操作等），现有框架无法复用。

目标是将教学协议从 Git 领域解耦，输出为一个 opencode skill，在任何仓库中加载该 skill 即可开启"想学什么就学什么"的交互教学。

## Goals / Non-Goals

**Goals:**
- 创建通用教学 skill `teach-anything`，不依赖任何领域知识
- 实现"用户诉求 → 初定方案 → 讨论 → 确认 → 教学"五步协商流程
- 保持 7 步教学流程（概念讲解 → 原子任务 → 结果审计 → 即时反馈 → 危机模拟 → 结课测验 → 灵魂追问）
- 教学计划由 skill 在对话中动态生成，无需预写文件
- 兼容现有 Git 模块作为"内容插件"复用

**Non-Goals:**
- 不修改现有 Git 教学模块（module-0 至 module-6）
- 不创建 Web UI 或外部服务
- 不实现学习进度持久化存储
- 不支持多用户并发

## Decisions

| 决策 | 选项 | 选择 | 理由 |
|------|------|------|------|
| 实现形式 | Skill vs MCP vs 独立脚本 | **Skill** (SKILL.md) | 教学本质是对话交互，skill 直接嵌入 Agent 对话流，无需独立服务 |
| 内容来源 | 预写 files vs 动态生成 | **动态生成** | 未知主题无法预写内容，AI 实时生成模块计划和练习 |
| 教学结构 | 自由探索 vs 结构化模块 | **结构化模块** | 保持 7 步流程的纪律性，确保学习深度 |
| 内容存储 | 持久化 vs 会话级 | **会话级** | 教学计划仅存于当前对话，简化实现 |
| 语言 | 中英双语 vs 纯中文 | **纯中文** | 目标用户群体为中文零基础学习者 |

## Architecture

```
用户输入 "我想学 X"
  │
  ▼
[Phase 1: 学习方案协商]
  1. 解析诉求 → 提取主题、目标、前置要求
  2. 生成方案 → 分解为 N 个模块，每个模块有标题和目标
  3. 展示方案 → 呈现给用户审阅
  4. 用户反馈 → 调整模块/顺序/深度
  5. 确认方案 → 锁定教学计划

  │
  ▼
[Phase 2: 逐模块教学]
  for each 模块 in 教学计划:
    1. 概念讲解 → 用例子解释核心概念
    2. 原子任务 → 给出可执行的小练习
    3. 结果审计 → 验证练习结果
    4. 即时反馈 → 分析正确/错误
    5. 危机模拟 → 制造常见错误场景
    6. 结课测验 → 1-3 道综合题
    7. 灵魂追问 → 2-3 个开放式思考题
  end
```

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 动态生成内容质量依赖底层模型 | 用 7 步教学流程约束生成结构，确保教学深度 |
| 无预写练习 = 无法自动化验证 | 验证步骤依赖 Agent 的自主判断（如 `pip install` 成功与否） |
| 开放领域可能偏离主题 | 方案协商阶段锁定 scope，教学阶段严格执行计划 |
| 用户目标极度模糊（"我想学编程"） | Skill 通过追问缩小范围，直到可分解为具体模块 |

## Open Questions

- 学习方案应存为临时文件还是纯对话记忆？临时文件便于复查但增加复杂度
- 危机模拟步骤在通用教学中如何实现？Git 有 `git reset --hard` 等可操作场景，抽象领域可能无法模拟
- 是否需要支持"中途切换话题"？
