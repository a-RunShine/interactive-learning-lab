# 模块 1 错误分析

---

## 问题 1：漏了步骤 —— 没修改文件就 `git diff`

**操作：**
- 做完 commit 后直接执行 `git diff`
- 漏掉了"修改 practice.txt"这一步

**追问：** 为什么会跳过第 4 步？

**回答：** 没看清题目要求，有点粗心。不知道 `git diff` 是什么效果。

---

## 问题 2：`-- staged` vs `--staged`

**操作：**
```bash
git restore -- staged temp.txt
# 多了空格！
```

**报错：**
```
error: pathspec 'staged' did not match any file(s) known to git
```

**根因：**
- 不知道 `--staged` 是什么意思
- 不知道 `--` 和后面的词之间不能加空格

**知识点：**
- `--staged` 是一个整体 flag，意思是"对暂存区操作"
- `git restore <文件>` = 撤销工作区改动
- `git restore --staged <文件>` = 撤销暂存区改动

---

## 问题 3：`git restore` 忘了给文件参数

**操作：**
```bash
git restore
# fatal: you must specify path(s) to restore
```

**根因：** 不知道 `restore` 是什么意思

**知识点：**
- restore = 恢复、还原
- 命令格式：`git restore <文件>`
- 类比：`git add` 添加、`git commit` 提交、`git restore` 还原

---

## 问题 4：三次 `git diff` 无输出才意识到问题

**操作：**
```bash
git diff              → 空
git restore --staged  → 报错
git diff --staged     → 空
```

**根因：** 不知道 `git diff` 空输出是不正常的，不知道 `git status` 是诊断工具

---

## 根因总结

### 1. 不知道命令的含义就执行

- `git diff` — 不知道是什么
- `git restore --staged` — 不知道 `--staged` 是什么
- `git restore` — 不知道 `restore` 是什么意思

**后果：** 无法判断输出是否正常，无法预判会不会报错。

### 2. 不理解时不停下来诊断

- 连续三次无意义的命令执行
- 不知道 `git status` 是万能诊断工具

**后果：** 盲目尝试，浪费时间和精力。

### 3. 不核对当前状态就跳步

- 做完一步急着做下一步
- 没确认结果就继续

**后果：** 错误累积，到最后才发现问题。

---

## 改进方向

| 习惯 | 为什么重要 |
|------|-----------|
| 先搞懂命令再执行 | 知道什么是正常输出 |
| 遇到困惑先 `git status` | 诊断当前真实状态 |
| 做完一步确认结果 | 及时发现问题 |

---

## 核心原则

```
遇到 Git 问题 → 第一反应：git status
```

`git status` 是 Git 的"体温计"，万能诊断工具。
